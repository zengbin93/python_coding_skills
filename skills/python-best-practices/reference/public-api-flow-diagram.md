# 公共 API 内部调用流程图编写指南

当模块包含**公共 API 入口函数**且其内部调用链涉及多个私有辅助函数（典型如批处理流水线、多阶段工作流、回测/训练入口）时，**必须**在模块顶部 docstring 中绘制 ASCII 调用流程图。这是提升代码可读性最有效的手段之一：读者无需逐行追踪即可建立完整心智模型。

## 何时需要绘制

- ✅ 公共 API 内部调用 ≥ 3 个私有辅助函数
- ✅ 存在嵌套循环或多级分发（如 `for problem: for model:`）
- ✅ 涉及数据加载 → 转换 → 计算 → 落盘等多阶段流水线
- ✅ 输出会产生具有结构的目录或文件树
- ❌ 单一函数、简单 CRUD、纯工具函数无需绘制

## 编写规范

1. **位置**：放在模块 docstring 内，紧跟一句话功能说明之后
2. **入口签名**：第一行写出入口函数完整签名，让读者立刻看到参数
3. **分节标号**：用 `§1`、`§2.1`、`§2.2` 等标号对应代码中的注释，使图与实现一一对应
4. **缩进表达层级**：用 `│`、`├──`、`└──` 表达调用层级和顺序
5. **关键产物**：在右侧用箭头 `→` 标注每个函数的产物（文件、DataFrame、配置）
6. **同步配套**：代码中用 `# ═══ §N 章节标题 ═══` 分节注释呼应流程图
7. **附加目录树**（可选）：若产生文件输出，在流程图后追加目录结构示意

## 维护原则

- **同步更新**：重构调用链时，**同步**更新流程图。流程图与代码不一致比没有流程图更糟糕
- **简洁优先**：每行尽量不超 80 字符，过深嵌套用子图拆分
- **保持纯文本**：使用 ASCII 树形分支 `├── └── │` 即可，避免复杂 Unicode 框线字符
- **配套分节注释**：代码中的 `# ═══ §N ═══` 章节标题与流程图标号必须严格对应，方便读者在图与代码间跳转
